report writer 报告生成工作流

从用户提交 30 道题答案，到 LLM 流式输出人格报告，再到前端实时分段渲染——完整数据流向、字段来源、协议格式一图看懂。

01五阶段总览

STAGE 1
/quiz/submit
scoring engine 算分 + Enrich 阶段从 5 张字典表注入静态知识
→ diagnosis dict
STAGE 2
DB 落库
assessment.diagnosis_json 持久化
status = analyzed
→ DB row
STAGE 3
/ws/result
前端连 WS，后端取出 diagnosis_json 调度 report writer
→ system + user msg
STAGE 4
豆包 LLM
temperature=0.6 流式输出含 --Section-- 标记的纯文本
→ raw text chunks
STAGE 5
切片 & 落库
_SectionStreamer 实时切片推 WS，写库 report_text
→ status = complete

03report writer 输入：System Prompt + User Message

SYSTEM PROMPT（静态文件）
# 来源
docs/agent-b-system-prompt.md

# 加载方式（启动时一次）
AGENT_B_SYSTEM_PROMPT =
  _PROMPT_FILE.read_text()
    .split("\n## 版本记录")[0]
    .rstrip()

# 内容性质
- 产品写的写作规则长文
- 规定 --Section-- 标记协议
- 段落字数 / 语气 / 句式风格
- 追逃视角 / aligned=false 特殊处理
USER MESSAGE（每次请求动态拼装）
# 用户类型
- 类型代码：MA-CL-H
- 类型名（用作报告标题）：温柔的守望者
- 副标题：你的稳定是底气，不是回避
- 类型锚定句（开篇必须以此展开）：<type_anchor>

# 五维度结果（D1-D5 全部展开）
- D1 依恋：interp = moderate_anxious
- D2 边界：interp = clear
- D3 冲突：interp = healthy，追逃角色 = pursue
- D4 爱的语言：top2 = ['T1', 'T2']
    · T1 言语肯定：<detail>
    · T2 精心时刻：<detail>
  aligned=false：用户主动选 T3，必须追加盲区段
- D5 表达风格：直爽热情型（高直接×高分享）
    · 该象限写作方向：<D5_guide>

# 深层洞察素材（每条 150-250 字）
1. 标题：压力表达崩塌 | severity=moderate
   写作种子：<report_seed>
   解读路径：<interp_path>
2. ...

由 app/agents/report_writer.py · build_user_message(diagnosis) 拼装。把字典表静态知识切片注入 prompt——LLM 不需要扫全表。

04LLM 输出格式：--Section-- 标记协议

--Title--类型标题
--Opening--开篇画像
--Attachment--依恋段(D1)
--Boundary--边界段(D2)
--Conflict--冲突段(D3)
--Language--爱语段(D4)
--Style--表达风格(D5)
--Highlight--深层洞察
--Suggestion--收尾建议
豆包返回（流式 chunk 拼接后）
--Title--《温柔的守望者》

--Opening--你不是没有边界感的人，你只是把边界画得离自己更近...

--Attachment--当对方沉默或冷淡时，你的依恋系统会被激活...

--Boundary--...

--Conflict--在冲突中你倾向于扮演"追"的角色...

--Language--你最被打动的两种爱的语言是言语肯定和精心时刻...

--Style--你的表达风格属于直爽热情型...

--Highlight--
High_1: 压力表达崩塌——日常小事你说得清楚，但...
High_2: 追逃循环（追方）——他越冷你越靠近...

--Suggestion--明天起，试着把"先确认他是不是生气"
换成"先确认我自己是不是太累了"...

由 system prompt 强制约束格式。后端 _SectionStreamer 用正则 /--([A-Za-z]+)--/ 实时识别标记，把 chunk 流切成 typed 消息推前端。

05WebSocket 消息时序（前端边收边渲染）

+0msmetatype_code / type_name / dim_chart / highlights_meta / segment_decode (一次性下发，前端立即渲染人格卡)
+180mssection_startsection: "Title"
+220mssection_chunksection: "Title", text: "《温柔的"
+260mssection_chunksection: "Title", text: "守望者》"
+280mssection_endsection: "Title"
+290mssection_startsection: "Opening"
…section_chunksection: "Opening", text: "你不是没有边界感..."
…section_endsection: "Opening"
…section_startsection: "D1" → "D2" → ... → "Highlight" → "Suggestion"
+12sdonepersonality_type / report_json (报告生成完毕)

前端 report.js 按 section 字段路由到对应数据槽（sec.Title / sec.Opening / sec.D1 ...）边收边渲染，不等全部到齐。

06Assessment 状态流转 & 落库字段

① pendingINITIAL

user_id · session_id
answers_json: null
diagnosis_json: null
report_text: null

② analyzedAFTER AGENT A

answers_json: [...30 题答案...]
diagnosis_json: {type_code, dims, highlights, ...}
report_text: null

③ generatingAGENT B RUNNING

status: "generating" (乐观锁防并发)
条件 UPDATE：仅当 status=analyzed 才能切到 generating
失败回滚：异常时 status 重置回 analyzed 允许重试

④ completeFINAL

report_text: "--Title--...--Suggestion--..."
report_json: {"raw_llm_output": "..."}
personality_type: "MA-CL-H"
后续请求直接命中缓存，不再调 LLM
